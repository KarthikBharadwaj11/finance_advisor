The financial advisor is an AI-enabled applications for users to get financial suggestions, inputs and advice from. The app mainly consists of 6 layers, 
The User request, the API layer, the service layer, the agentic layer, the tools layer, the RAG layer and the DB layer. When the suer asks a question, the FASTAPI validates the request body against the request schema. The middleware assigns a request Id and the latency timer is started. The get_db dependency then injects the db session where the query checks I the user exists or not. If the user doesn't exit, a 404 error is sent out.  The gent loop the starts running the executor.py. Here the LLM is called and it decodes where a tool is required and needs to be called. The tool is then executed, it runs the fiery, calculates metrics, checks FAISS, which is one of the data sources, and retunes back a response. The result is fed back to the LLM for each iteration. There are a limited number of iterations to avoid an infinite loop and the final output is the nest back in a structured format. The fastAPI validates the response with the response Schema. The middleware then logs the time taken and the request ID. The response is then sent to the user. 
The 2 main data sources we have are FAISS and postgreSQL. Keep in mind that the LLM doesnt access the db directly at any point. The backend services query required data and feed it to the LLM since LLMS are text-in and text out. So they cannot handle or query data. 
FAISS stored all generic knowledge related to finance, such as investment 101, budgeting principles etc. postfgreSQL stored user related information thats specific to each user. Both of these are required or the LLM to preduce a valid response. 
We have 5 tools in our app that make this happen, and all are interdependent of each other. The order in which they are called matters for a valid output.  This entire operation is overseen by prompts. prompts are important because it gives the LLM a personality, a content of whats happening and how the LLM should behave. 
Prompt engineering have 4 major rules, which are to never give advices from memory and to always call tools, to never give out individual stock, options remommendations and to always give out assist classes and index funds recommendations, to always retrieve knowledge about the topic for a given query before giving out a response, and lastly, to always ask for clarifications if the given information isnt enough to give a suggestion. The first and the third are very important and are foundation rules.

Why grounding is important?
Without the grounding rules the LLM might answer very vaguely. This result in hallucinations. For example, if the user asks the system if his savings this month are enough to buy a car, the LLM without context can reply saying his savings are $2000 this month and he cannot afford a car. Without his financial knowledge, the number is just made up. The user has never mentioned any numerical value to the LLM for it to come up with $2000. Whereas with context, the LLM will know what the user's savings are and will give out a valid response.

When the iteration limit is hit, my app returns 'Analysis incomplete: maximum reasoning steps reached' rather than a partial result or a silent failure. It's a graceful degradation — the user knows something went wrong rather than receiving a fabricated or incomplete analysis presented as complete. In a production improvement, I'd also want to return whatever partial tool results were collected before the limit was hit, so the user gets some value even from an incomplete run
